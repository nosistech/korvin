# KORVIN Configuration

This page explains the public KORVIN configuration surface.

KORVIN currently has two configuration categories:

1. Public example files that are safe to inspect, commit, and package.
2. Private local secret files that users create later.

Keep these categories separate. Public examples must never contain real provider keys, bot tokens, passwords, private URLs, VPS details, or personal secrets.

## Configuration Principles

- Keep defaults local-only.
- Keep secrets out of Git.
- Keep provider keys in environment variables.
- Keep example files safe to inspect and share.
- Keep runtime services separate from local setup files.
- Keep model choice configurable, not hardcoded.
- Document only what works today.

## Files Generated by korvin init

A typical local setup generated by `korvin init` includes:

```text
.env.example
.gitignore
korvin.config.json
README.local.md
config/providers.example.json
config/litellm.example.yaml
config/dashboard.example.json
docs/SETUP_SUMMARY.md
docs/NEXT_STEPS.md
docs/SECURITY_NOTES.md
data/
memory/
logs/
backups/
```

If `--voice` is used, KORVIN also prepares placeholder voice folders and an example voice profile.

These files are preparation only. They do not install production services or configure public internet access.

## .env.example

`.env.example` is a safe template. It may include placeholder variables such as:

```text
KORVIN_PROVIDER_KEY=
KORVIN_DASHBOARD_KEY=
LITELLM_MASTER_KEY=
```

Rules:

- `.env.example` may be committed.
- `.env.example` must not contain real secrets.
- Users should copy `.env.example` to `.env` only when ready to add local secrets.
- Real secrets belong in `.env`, not in example files.

## .env

`.env` is the private local secrets file.

Rules:

- Do not commit `.env`.
- Do not paste `.env` into chats, issues, screenshots, public docs, or support requests.
- Rotate any secret that was accidentally exposed.
- `korvin doctor` may detect whether `.env` exists, but it does not read it.

## korvin.config.json

`korvin.config.json` records local setup state.

Safe local defaults include:

```json
{
  "profile": "quick-local",
  "dashboard": {
    "enabled": true,
    "mode": "local-only",
    "host": "127.0.0.1",
    "port": 3002
  },
  "security": {
    "internetExposure": false,
    "publicPortsConfigured": false,
    "secretsWritten": false
  }
}
```

Important defaults:

- Dashboard host is `127.0.0.1`.
- Internet exposure is `false`.
- Public ports configured is `false`.
- Secrets written is `false`.
- Provider is not configured by default.

Do not change these values casually. Remote access should be configured only through a protected deployment path.

## config/providers.example.json

`config/providers.example.json` is a provider placeholder file.

Example pattern:

```json
{
  "provider": "openai-compatible",
  "model": "your-model-name",
  "baseUrl": "https://api.example-provider.com/v1",
  "apiKeyEnv": "KORVIN_PROVIDER_KEY"
}
```

Rules:

- Replace placeholders only in private local copies.
- Do not place real provider keys in this file.
- Use environment variables for provider keys.
- Keep provider choice configurable.
- Do not hardcode one model provider as the identity of KORVIN.

## config/litellm.example.yaml

`config/litellm.example.yaml` is a future LiteLLM routing example.

Example pattern:

```yaml
model_list:
  - model_name: korvin-default
    litellm_params:
      model: openai/your-model-name
      api_key: os.environ/KORVIN_PROVIDER_KEY
```

Rules:

- Keep API keys in environment variables.
- Do not commit real LiteLLM secrets.
- Do not claim LiteLLM runtime setup is automatic in `korvin init`.
- Keep routing model-agnostic.

## config/dashboard.example.json

`config/dashboard.example.json` describes local dashboard defaults.

Safe default:

```json
{
  "enabled": true,
  "host": "127.0.0.1",
  "port": 3002
}
```

Rules:

- Keep dashboard examples local-only by default.
- Do not expose raw dashboard ports directly to the public internet.
- Use a tunnel and access-control layer for protected remote access.

## Repair Mode

Running `korvin init` again on an existing setup enters repair mode.

Repair mode may restore missing generated files.

Repair mode does not overwrite existing user files.

Important preserved files:

- existing `.env`
- existing `korvin.config.json`
- existing user-created files
- existing generated files that are already present

After repair, run:

```bash
korvin doctor <folder>
```

A healthy setup should show:

```text
Final status: Ready
```

## What korvin init Does Not Configure

`korvin init` does not configure:

- VPS setup
- Cloudflare setup
- Telegram setup
- LiteLLM runtime setup
- Dashboard service setup
- systemd services
- Open WebUI
- Provider keys
- Public ports
- Production deployment
- Voice runtime dependencies

This is intentional. The current npm CLI creates safe local setup files and checks only.

## Secret Handling Rules

Never put real secrets in:

- README files
- public docs
- screenshots
- GitHub issues
- GitHub releases
- npm package files
- example config files
- AI chat transcripts
- terminal output shared publicly

Use placeholders instead:

```text
<REDACTED_SECRET>
<VPS_IP>
<SSH_USER>
<INTERNAL_PATH>
```

## Recommended User Flow

```bash
npm install -g @nosistech/korvin@latest
korvin init ./korvin-local
korvin doctor ./korvin-local
cd ./korvin-local
```

Then review:

```text
README.local.md
docs/SETUP_SUMMARY.md
docs/NEXT_STEPS.md
docs/SECURITY_NOTES.md
korvin.config.json
config/providers.example.json
config/litellm.example.yaml
config/dashboard.example.json
```

Create `.env` only when ready to add local secrets.
